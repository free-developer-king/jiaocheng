基础一：Codex 是什么，以及该用哪个入口

目标：理解 CLI、桌面 App、IDE、Web/Cloud 的差异，知道什么时候用本地，什么时候交给云端。

你要掌握

  • Codex 是会读文件、改文件、跑命令、复查结果的 coding agent，不只是聊天助手。
  • CLI 适合本地仓库和终端工作；桌面 App 适合长任务、网页验证、文件协作；IDE 适合贴着编辑器改代码。
  • 安全默认值要保守：新项目先用工作区写权限，明确需要时再放宽。

基础二：提示词、计划、验证闭环

目标：把“我想要”变成 Codex 可执行的目标、上下文、约束和完成标准。

你要掌握

  • 四段式提示词：Goal、Context、Constraints、Done when。
  • 复杂任务先计划，简单任务直接执行，避免把所有任务都变成会议。
  • 让 Codex 自己跑验证：测试、lint、截图、页面点击、diff review。

案例：做一个个人任务看板网页

Goal: 做一个单文件个人任务看板网页。 Context: 当前目录为空，优先使用原生 HTML/CSS/JS。 Constraints: 不引入构建工具；移动端可用；按钮文字不能溢出。 Done when: 页面可打开，能新增任务、切换状态、筛选待办/完成，刷新后数据仍在。

  • 页面在手机宽度下不重叠、不遮挡。
03

进阶一：AGENTS.md、MEMORY.md 与项目约束

目标：把反复说的工作方式写成项目规则，让 Codex 每次进场就知道边界。

你要掌握

  • `AGENTS.md` 放团队规则、验证命令、禁区和完成定义；越靠近子目录的规则越具体。
  • `MEMORY.md` 适合记项目决策、踩坑、资源位置；凭据只记位置不记值。
  • 规则不是越多越好，真正重复两三次的错误才值得沉淀。
进阶AGENTS.md记忆

案例：为你的 AI Native 工作区建立规则系统

请审计当前工作区，并建立 AI Native 工作规则系统。 先读现有 AGENTS.md / MEMORY.md。 然后补充： 1. 项目启动检查流程。 2. Codex 修改文件前后的验证要求。 3. MEMORY.md 的新增规则和禁止记录内容。 4. 一个可复用的四段式提示词模板。

验收标准

  • 新线程打开项目后，Codex 能主动读规则和记忆。
  • 规则能减少重复沟通，而不是制造额外负担。

进阶二：MCP、插件与外部工具连接

目标：让 Codex 访问文档、浏览器、GitHub、Figma、Google Drive、Slack 等外部上下文。

你要掌握

  • MCP 是给 Codex 增加工具和上下文的协议，配置在 `config.toml` 或通过 `codex mcp` 管理。
  • 插件是更完整的分发单元，可以打包技能、应用连接和 MCP 服务。
  • 外部工具要按最小权限接入，尤其是浏览器、Drive、Slack、GitHub。

进阶三：Skills、非交互模式与自动化

目标：把稳定流程封装成 Skill，把重复检查交给 `codex exec` 或 Codex 自动化。

你要掌握

  • Skill 适合可复用流程：写作、审查、发布、报表、视频脚本、资料清洗。
  • `codex exec` 适合脚本、CI、日志总结、结构化输出和流水线。
  • 自动化适合周期性任务，但默认权限要谨慎，先手动跑通再定时。

01

基础篇

认识 WorkBuddy：从工具到桌面智能体

你要掌握什么

WorkBuddy 的本质是“能规划、能操作本地文件、能交付结果”的桌面智能体。学习第一章的目标不是熟悉每个按钮，而是知道什么时候该用普通任务、什么时候该用助理、什么时候要指定工作目录。

  • 主界面由侧边栏、对话区、右侧结果区组成。
  • 任务可以处理文档、表格、PPT、数据、代码、设计和批量文件。
  • 工作目录决定它能读取和写入哪些项目文件。

案例：第一次让 WorkBuddy 交付文件

准备一个文件夹 `WorkBuddy练习/01-会议纪要`，放入一份会议录音转写或会议纪要文本。

请读取当前文件夹中的会议纪要，整理为一份可交付的行动清单。
要求：
1. 按“已确认事项 / 待确认问题 / 负责人 / 截止时间”输出表格；
2. 标出信息来源，不要补充纪要中没有的信息；
3. 生成 Markdown 和 docx 两个版本；
4. 完成后告诉我文件保存在哪里。

02

基础篇

任务闭环：创建、追问、查看结果、验收

任务描述四件套

一个可执行任务通常包含：目标、输入、约束、输出格式。WorkBuddy 可以持续追问，你也应该把修改意见留在同一个任务里，让它继承上下文。

  • 目标：让它知道最终要交付什么。
  • 输入：文件、截图、链接、知识库资料或文字说明。
  • 约束：字数、风格、格式、不要做什么。
  • 验收：如何判断结果能不能直接用。

案例：把销售 CSV 做成分析报告

请分析当前文件夹中的销售数据 CSV，生成一份管理层能看懂的分析报告。
输出要求：
1. 先检查字段含义和数据质量；
2. 找出销售额、客单价、转化率、复购率的关键变化；
3. 至少生成 3 张图表，并解释每张图的结论；
4. 输出 Excel 数据透视表、Markdown 分析报告和一页 PPT 汇报摘要；
5. 在最后列出需要我人工确认的数据口径。

验收清单

官方：任务对话 官方：结果查看 官方：任务管理
03

进阶篇

上下文工程：文件、截图、ima 知识库和记忆

为什么先做知识库

重复任务的瓶颈不是“不会写提示词”，而是每次都重新解释背景。ima 知识库适合沉淀选题、客户资料、项目文档、风格画像和复盘报告。

  • 知识库按任务目标建，不要一开始做大而全的资料仓库。
  • 第一批资料控制在 10-20 条，优先来源清楚、近期有效、内容完整。
  • 有复用价值的产物要回存知识库，形成知识复利。

案例：搭建“AI 工具教程素材库”

请基于我选择的 ima 知识库资料，帮我整理一份“AI 工具教程素材库”的结构建议。
输出：
1. 推荐知识库名称；
2. 一级分类和每类应收集的资料；
3. 哪些资料适合直接保存，哪些需要人工核对；
4. 后续每次生成教程后，哪些产物应该回存；
5. 给出一份可复制的“回存 ima 前检查清单”。

验收清单

官方：ima 功能指引 官方：知识复利技巧

进阶篇

专家、专家团和技能：让 AI 既懂行又能动手

三种能力怎么选

专家解决“用什么视角判断”，专家团解决“复杂任务如何分工”，技能解决“让 AI 具备具体工具能力”。复杂任务先找专家团，涉及外部操作再启用必要技能。

  • 专家：单点专业判断，比如法务审阅、营销策略、产品分析。
  • 专家团：多角色协同，比如选题、写作、配图、改编、排版。
  • 技能：封装脚本或工作流，比如发送邮件、读写文件、调用第三方 API。

案例：让专家团评审一份增长方案

请召唤适合的营销/增长专家团，评审当前文件夹里的《增长方案.md》。
请从以下角度输出：
1. 目标用户是否清晰；
2. 渠道策略是否匹配预算和资源；
3. 内容节奏是否能持续执行；
4. 数据指标是否能验收；
5. 风险、缺口和优先修改建议。
请最终生成一份“评审报告”和一份“修改版方案大纲”。

验收清单

官方：专家 官方：技能

进阶篇

连接器、自动化和远程助理：把重复工作变成系统

从“用一次”到“每天自动跑”

连接器把外部数据接进来，自动化按时间触发，助理让你在手机上远程下达任务。它们适合日报、周报、资料汇总、邮件筛选、会议纪要和内容选题。

  • 连接器支持邮箱、腾讯文档、腾讯会议、TAPD 等服务和自定义 MCP。
  • 自动化任务包含名称、工作空间、提示词、模型/技能、定时规则和推送设置。
  • 远程助理适合临时触发，但电脑要保持开机并运行 WorkBuddy。

案例：每天 8:30 自动生成工作简报
